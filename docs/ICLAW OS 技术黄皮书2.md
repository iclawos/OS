过去六十年，计算机世界建立在**字节**的确定性之上。LLM的崛起带来范式转移：基本单位从字节变为**token**，计算从确定性变为概率性，架构从“CPU+内核”变为“LLM+Agent”。ICLAW OS 是从内核到用户界面彻底拥抱这一新范式的**AI原生操作系统**。本黄皮书阐述其设计哲学、核心架构、信息流模型及工程路径。

## 1. 范式转移：从字节到Token

> “LLM = CPU，Agent = 操作系统内核。” —— Andrej Karpathy

传统计算机：CPU处理**字节** → 内核调度进程 → 操作系统服务用户  
AI时代：LLM处理**token** → Agent编排任务 → AI OS服务用户

| 维度 | 字节时代 | Token时代 |
| - | - | - |
| 基本单位 | 字节（确定性） | token（概率性） |
| 计算方式 | 精确、确定 | 统计、概率 |
| 瓶颈 | CPU主频、存储速度 | 上下文窗口、token传输 |
| 状态存储 | 分散于客户端/服务端 | 集中于本地Agent内核 |


ICLAW OS 的核心判断：**大模型运行在云端或集群，本地只运行智能体调度逻辑**。本地不再是算力中心，而是**智能体指挥中心**。


## 2. 设计哲学与架构总览

ICLAW OS 的设计围绕**一个基础、两大引擎、五层架构**展开：

### 2.1 一个基础

**LLM = CPU**：云端或集群中的大模型提供统计推理能力，是系统的“处理器”。  
**Agent = 内核**：本地智能体内核负责任务编排、上下文管理、工具调度。

### 2.2 两大引擎

- **上下文/记忆引擎**：管理Token化的短期/长期记忆，支持分层存储与高效检索。

- **工具/技能引擎**：将文件、网络、设备等能力抽象为原子工具，供Agent调用。

### 2.3 五层架构

```
┌─────────────────────────────────────────┐  
│           交互层（AI Shell / UI）         │  
├─────────────────────────────────────────┤  
│           Agent编排层（内核主循环）        │  
├─────────────────────────────────────────┤  
│     上下文/记忆引擎  │  工具/技能引擎       │  
├─────────────────────────────────────────┤  
│          设备抽象层（多模态驱动）          │  
├─────────────────────────────────────────┤  
│          硬件层（CPU/RAM/网络）           │  
└─────────────────────────────────────────┘
```


## 3. 内核设计：AI驱动的单线程主循环

借鉴Claude Code的**单线程主循环（Single-Threaded Master Loop）** 设计，ICLAW OS内核是一个事件驱动的AI调度器：

```
while (true) \{  
    task = fetch\_next\_task();        // 从Token队列获取任务  
    context = load\_context(task);    // 加载相关上下文窗口  
    result = call\_llm(context);      // 调用云端LLM推理  
    if (result.has\_tool\_calls()) \{  
        execute\_tools(result.tool\_calls);  
    \} else if (result.is\_complete()) \{  
        output\_result(result);  
    \} else \{  
        push\_subtask(result.next\_step);  
    \}  
\}
```

内核不运行大模型，只运行**Agent逻辑**，负责：

- 任务队列管理（优先级、依赖、并发）

- 上下文窗口的分配与回收

- Token流的压缩与解压

- 工具调用的执行与权限检查


## 4. 两大引擎：上下文/记忆引擎与工具/技能引擎

### 4.1 上下文/记忆引擎

- **三级记忆系统**（参考OpenClaw）：

  - L0：实时工作区（当前对话，毫秒级）

  - L1：Agent缓存（最近会话，秒级）

  - L2：长期存储（向量数据库，毫秒级检索）

  - L3（扩展）：云端历史压缩存储

- **共享Prompt缓存**：借鉴Claude Code的`subagent` fork模式，多个Agent可共享已计算的KV缓存。

- **自修复内存**：周期性检测“上下文熵”，自动压缩或遗忘低价值Token。

### 4.2 工具/技能引擎

- **插件化工具体系**：每个工具是一个独立的原子能力（读文件、执行bash、HTTP请求、LSP补全等）。

- **工具注册与发现**：内核维护工具清单，支持动态加载。

- **标准工具接口**：

- ```
interface Tool \{  
  name: string;  
  description: string;  
  parameters: JSONSchema;  
  execute(input: any): Promise\<TokenStream\>;  
\}
```

- **工具调用即系统调用**：Agent通过标准化ABI调用工具，权限由内核统一管理。


## 5. 进程与调度：Agent为一等公民

### 5.1 Agent进程模型

每个Agent是一个**自主执行实体**，拥有：

- 唯一ID

- 独立的上下文窗口预算

- 优先级和调度策略

- 状态（就绪/运行/等待工具/休眠）

### 5.2 智能调度器

借鉴SchedCP等研究，调度器由一个小型LLM驱动：

- **动态学习工作负载特征**（任务类型、token消耗、延迟敏感度）

- **智能分配“计算预算”**（即LLM调用频率和上下文窗口大小）

- **实现应用感知**：前台交互Agent优先，后台数据分析Agent在空闲时执行


## 6. 内存与上下文管理：分层Token存储

Token时代的“内存”就是**上下文窗口**。ICLAW OS实现分层上下文架构：

| 层级 | 名称 | 介质 | 典型大小 | 延迟 |
| - | - | - | - | - |
| L0 | 实时工作区 | RAM | 8K-32K tokens | \<1ms |
| L1 | Agent缓存 | RAM | 32K-128K tokens | \<10ms |
| L2 | 长期存储 | 本地向量库 | 无限制 | \<100ms |
| L3 | 云端扩展 | 云端存储 | 无限制 | 1s+ |


**上下文压缩协议**：当L0+L1接近上限时，内核请求云端LLM将一段历史Token压缩为**摘要Token**（一个特殊Token代表整段内容）。摘要Token可嵌套，形成**Token树**。


## 7. 设备与I/O：多模态Token化驱动

### 7.1 多模态驱动模型

传统驱动提供字节流（音频PCM、图像RGB），ICLAW OS驱动提供**Token化的感知信息**：

- 摄像头驱动 → 输出物体/场景的Token描述

- 麦克风驱动 → 输出语音识别的Token序列

- 触控板驱动 → 输出手势意图的Token

### 7.2 跨设备统一调度

通过系统级API，Agent可透明访问所有已连接设备（手机、平板、IoT）。例如：

```
agent.call\_device("my\_phone", "camera.capture", \{"query": "拍下桌上的文档"\})
```

返回的是Token化的文档内容，而非图像字节。


## 8. 安全模型：护栏即系统调用

### 8.1 内核级护栏

所有Agent执行的工具调用，必须经过**护栏引擎**审查：

- 白名单/黑名单规则

- 用户预设的安全策略（如“不允许删除文件”）

- 异常行为检测（如短时间内大量调用敏感工具）

### 8.2 细粒度权限控制

每个工具对应一个权限。Agent首次调用工具时，内核向用户弹出授权请求（类似Android权限模型）。用户可授予“一次”、“本次会话”或“永久”。

### 8.3 隐私设计

- 所有会话状态、上下文、用户偏好**仅存储在本地**

- 上传到云端的Token经过**脱敏**（如替换实体名、哈希敏感ID）

- 云端LLM无状态，不记忆任何用户数据


## 9. 信息流模型：Token环流

信息以**概率性Token流**在四个节点间循环：

```
          ┌─────────────┐  
          │  云端大模型   │ (统计推理)  
          └──────▲──────┘  
                 │ Completion Tokens  
                 │  
    ┌────────────┴────────────┐  
    │                         │  
Prompt Tokens            Prompt Tokens  
    │                         │  
    ▼                         │  
┌─────────┐               ┌───┴────┐  
│本地电脑  │◄────结构化数据──►│  网站  │  
│(Agent)  │   (工具调用)    │(工具场)│  
└────┬────┘               └────────┘  
     │ 意图/结果  
     ▼  
┌─────────┐  
│移动端APP│  
│(投影终端)│  
└─────────┘
```

**核心特性**：

1. **Token是唯一流通货币**：任何字节进入Agent后立即Token化。

2. **概率性贯穿全程**：LLM输出带权重，Agent在多个概率路径中选择。

3. **网站退化为工具场**：只提供API，不提供UI。

4. **移动端APP是Agent投影**：只做采集和渲染。

5. **本地是唯一状态持有者**：云端无状态。


## 10. 硬件规格建议

ICLAW OS本地客户机（Agent指挥中心）的推荐配置：

| 使用场景 | CPU | RAM | 存储 | GPU | 网络 |
| - | - | - | - | - | - |
| 学习/试用 | 2核 | 8GB | 20GB SSD | 不需要 | 20Mbps |
| 日常开发（单Agent） | 4核 | 16GB | 50GB SSD | 不需要 | 100Mbps |
| 复杂项目（多Agent并行） | 8核 | **32GB** | 100GB NVMe | 不需要 | 100Mbps+ |
| 生产级多Agent集群 | 8核+ | **64-96GB** | 200GB+ NVMe | 可选（本地小模型） | 500Mbps+ |


> 32GB是舒适基准线，因为多Agent同时运行时的KV缓存和上下文窗口叠加需求极高。


## 11. 工程实现路径

### 阶段一：内核原型（AIOS Kernel）

- 研究AIOS、OpenClaw、Claude Code（泄露代码）的架构

- 以Linux为基座，实现最小化Token任务队列 + 单线程主循环

- 实验：将Qwen-2.5-3B作为LKM集成到内核空间（可选）

### 阶段二：工具与协议

- 实现Gateway和插件化工具系统（参考OpenClaw）

- 定义AI原生应用ABI：Agent ↔ 内核的系统调用接口

### 阶段三：记忆与系统服务

- 集成本地向量数据库（如LanceDB、Qdrant）

- 实现三级记忆系统（L0/L1/L2）

### 阶段四：交互界面

- 开发AI Shell：自然语言命令、思维链可视化、任务管理面板

- 设计以“意图”为中心的UI范式

### 阶段五：安全与分发

- 实现内核级护栏引擎

- 构建Agent应用商店与沙箱机制


## 12. 总结与展望

ICLAW OS 不是对现有操作系统的修修补补，而是**从字节到Token的彻底范式转移**。它将：

- 大模型作为推理CPU（云端/集群）

- 智能体作为调度内核（本地）

- Token作为统一数据单位

- 概率性作为第一类公民

未来，每个用户的数字世界由一棵**私有Token树**构成，ICLAW OS是其根节点。我们诚邀开发者、研究者共同参与这一操作系统的重构。

> “底层的数据单位变了，上面的一切都得跟着变。” —— Andrej Karpathy
